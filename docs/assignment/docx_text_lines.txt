COMPENG 3510 Final Project, Option 3
AI-Assisted Development of a Modbus Client for the ABB REF620 ANSI Feeder Protection Relay
Version 3, Revised Project Description
1. Overview
This project requires each team to develop, using AI-assisted software development, a Windows-based educational Modbus client that reads a documented subset of operational data from the ABB REF620 ANSI feeder protection and control relay and performs a small, low-risk subset of control writes. The pedagogical objective is to ensure that students can connect the abstract structure of a Modbus TCP message to the specific vendor implementation documented in the ABB Modbus Communication Protocol Manual (documen
t 1MAC458836-IB) and the ABB Modbus Point List Manual (document 1MAC509789-IB
), and
 can empirically validate that connection with Wireshark packet captures. The secondary objective is to develop disciplined practices for AI-assisted engineering, including prompt design, output verification, documentation of human decisions, and citation of sources.
The 
deliverables
 of this option
 include
 a final report, a final presentation, and the complete set of document files produced during development
.
2. Protocol Scope and Addressing
The REF620 ANSI exposes Modbus over TCP (IANA port 502) and over serial (Modbus RTU or ASCII). Teams shall target Modbus TCP for all required work because packet capture is straightforward and the MBAP header simplifies framing analysis.
Addressing is restricted to two Modbus data areas, the 0X area (coils) and the 4X area (holding registers). The 1X discrete input area is excluded from the scope of this assignment. This restriction is consistent with the REF620 ANSI implementation because all monitoring data, including measurands, system status registers, event records, fault records, and real-time clock structures, is available through the 4X area, and because all control operations are performed through 4X control structures, with coil a
ccess providing the supplementary one-bit view. The addressing convention follows the Modbus PLC principle, in which the base address is 1 and application data resides in the range 1 to 9999.
3. AI-Assisted Development and Academic Integrity
AI-assisted development, colloquially termed vibe coding, denotes an iterative workflow in which a human engineer specifies intent and constraints to a generative model, inspects the model output, tests it against objective evidence, and revises the prompt or the code until the evidence is satisfied. The method is distinct from both manual coding and fully autonomous code generation because it requires explicit human judgment at every transition between prompt, artifact, and validation. Students 
in
 this course shall adopt this workflow deliberately and document it rigorously.
To uphold academic integrity, every team shall maintain a written prompt log. The log shall record the prompts that materially influenced design, implementation, debugging, or evaluation decisions; the corresponding AI output artifacts; and a short annotation indicating what the team accepted, corrected, or rejected, and why. AI-generated text that appears in the final report shall be labeled and accompanied by the team's own verification. Undisclosed reliance on AI output without human verification constit
utes a violation of academic integrity under the course syllabus.
4. Modbus Technical Primer
The following primer states the minimum technical foundation students require before formulating prompts. It is derived from the ABB REF620 ANSI Modbus manuals and summarizes only the elements necessary to complete this project. Students shall consult the ABB manuals for authoritative detail.
4.1 Transport and Framing
Modbus TCP frames begin with a Modbus Application Protocol (MBAP) header comprising a transaction identifier, a protocol identifier (zero for Modbus), a length field, and a unit identifier. The MBAP header is followed by the Protocol Data Unit (PDU), which consists of a function code and function-specific data. Serial variants (Modbus RTU and Modbus ASCII) omit the MBAP header and instead rely on a CRC or LRC trailer. This distinction is important when comparing a TCP capture to serial documentation.
4.2 Supported Function Codes in Scope
The REF620 ANSI supports application function codes 01, 02, 03, 04, 06, 08, 16, and 23, and a set of serial diagnostic subfunctions through function code 08. Because this project restricts addressing to 0X and 4X, only the following function codes are within scope.

Function code 02 (read discrete inputs) is out of scope because the 1X area is out of scope. The diagnostic functions accessible via function code 08 are out of scope for Modbus TCP in this project.
4.3 Data Areas within Scope
Two data areas are used. Coils (0X) carry single-bit indications and are read with function code 01. Holding registers (4X) carry both monitoring data and control structures and are read with function code 03 and written with function codes 06 or 16. The REF620 ANSI 4X map includes measurand registers, system status registers (SSR1 through SSR6 at addresses 9000 through 9005), real-time clock structures, event records beginning at 9040, fault records beginning at 8000, and control structures beginning at 10
00.
4.4 Register Width, Word Order, and Scaling
Most measurands are encoded in a single 16-bit register as an unsigned or signed two's-complement value. Selected measurands are encoded in two consecutive 16-bit registers forming a 32-bit signed integer; in these 
cases
 the high word occupies the lower register address and the low word occupies the next address. For example, the three-phase current measurement logical node CMMXU1 exposes 
the phase A
 current amplitude as a signed 32-bit value at register 2000, with the low word at register 2001, under scale factor 100. The engineering value is obtained by applying the vendor-specified scaling formula, namely engineering value equals (Modbus value minus offset) divided by scale factor, which is the inverse of the mappi
ng defined by ABB as Modbus value equals (IEC 61850 value multiplied by scale factor) plus offset.
4.5 Indication Semantics in the 4X Area
Digital indications are additionally available as bit-packed 16-bit registers in the 4X area. Each indication is represented by two adjacent bits, a momentary position bit (MOM) and a momentary change detection bit (MCD). The MCD bit is set when the underlying indication has changed two or more times since the last successful read by the requesting client, and it is 
cleared
 on read for that client only. Double-point objects such as circuit breakers also expose encoded open and close entities with fault detection. Teams reading indication data shall decode MOM and MCD pairs correctly and shall not interpret an MCD reset as a value change on subsequent reads.
4.6 Exception Codes
The REF620 ANSI returns three Modbus exception codes under the conditions indicated in the vendor documentation: 01 (illegal function), 02 (illegal data address), and 03 (illegal data value). Write requests that are syntactically valid but rejected by the relay for operational reasons return exception code 03, with an internal reason code available from System Status Register 6 (SSR6) for diagnosis.
4.7 Control Structures and Write Operations
Writes in this project are executed only through the 4X control structures documented in the ABB point list. Each control structure consists of an execute register, two password registers, a control register whose individual bits represent distinct actions, and a confirmation register. A 
valid write
 transaction sets the password registers, sets the desired action bit in the control register, and commits through the execute register, followed by verification of the confirmation register.
To preserve laboratory safety, this project restricts writes to low-risk actions that do not cause circuit-breaker operation. Acceptable write targets include the LED reset at control structure 1 bit 0 
(LD0.LLN
0.LEDRs1.Oper
.ctlVal) and the disturbance-recorder manual trigger at control structure 2 bit 0 (DR.RDRE1.RcdTrg.Oper.ctlVal). Circuit-breaker Select, Operate, Direct Open, and Direct Close commands are out of scope. Teams may discuss circuit-breaker control structures conceptually in the 
report, but
 
shall
 not issue such commands.
5. Project Requirements
5.1 Modbus Basic
 Read
Establish a Modbus TCP connection to the REF620 ANSI and reliably read at least one 16-bit measurand, for example a sub-register of CMMXU1 or a system status register.
Construct the read request using function code 03 and parse the response field by field.
Capture both 
request
 and response in Wireshark and annotate the MBAP header and PDU byte by byte.
Convert the raw value to an engineering value using the scale factor and offset from the point list.
Display the point name, the 4X address, the value type, the raw value, and the engineering value in a small Windows graphical interface.
Submit a clean, software-readable point table (CSV or XLSX) containing at minimum the columns: point name, function code, 0X or 4X address, register width, data type, scale factor, offset, units, access, and IEC 61850 reference.
5.2 Modbus 
Multiple Read and Write
Support multiple points selected by the user at runtime, with no hard-coded single-point limitation.
Correctly decode at least one 32-bit measurand, for example the CMMXU1 phase A current amplitude at register 2000, including two-register concatenation with high-word-first ordering and application of the scale factor.
Decode at least one MOM and MCD indication pair from the 4X bit-packed area and explain the semantics in the report.
Support both a single on-demand read and a configurable periodic read for the selected point set, with graceful handling of exception codes 01, 02, and 03.
Display the constructed request and the received response in hexadecimal within the application, together with a plain-language field decode.
Perform at least one bounded write to a low-risk control-structure target, specifically an LED reset or a disturbance-recorder manual 
trigger, and
 verify the outcome through the confirmation register and an independent observation.
6. Required Workflow
The workflow is staged to separate comprehension, construction, and empirical validation. Each stage has an explicit exit condition, and teams shall not advance without that condition being satisfied.
Explain before code. Ask the AI assistant to explain, field by field, the structure of the intended Modbus TCP read message. The exit condition is a written explanation that identifies every 
byte
 of the MBAP header and the PDU.
Prepare the point table. Extract the target points from the REF620 ANSI Modbus Point List Manual into a clean CSV or XLSX file that is 
software-readable
. The exit condition is a table whose schema is stable enough to drive the rest of the software.
Draft one read. Ask the AI assistant to generate Python code for Windows that issues one function-code-03 read for one selected point and parses the response. The exit condition is a demonstrated successful parse on a single sample.
Validate with Wireshark. Capture the request and the response, compare them to the designed message, and annotate any mismatch. The exit condition is a byte-exact match between the designed and captured request.
Expand the backend. Generalize the single-point read into a backend module that loads the point table, issues 
reads
 on demand and periodically, decodes 16-bit and 32-bit measurands, interprets MOM and MCD pairs, and applies scaling. The exit condition is a backend that supports the Enhanced read requirements.
Add a bounded write. Implement one LED reset or one disturbance-recorder trigger using the 4X control-structure pattern, verify the confirmation register, and capture the write transaction in Wireshark. The exit condition is a successful, documented 
write
.
Build the graphical interface. Layer a small Windows GUI over the validated backend, exposing point selection, read controls, the bounded write, the engineering and raw values, and the request and response inspection panel.
7. Starter AI Prompt Templates
The following templates are intended as starting points and shall be refined by the team as evidence accumulates. Each template assumes a Windows host running Python.
7.1 Explain the message
"Explain the structure of a Modbus TCP read request that uses function code 03 to read one 16-bit holding register from an ABB REF620 ANSI relay. Describe every byte in the MBAP header and in the Protocol Data Unit. Show one concrete example. Do not write code yet."
7.2 Build the point table
"Using the ABB REF620 ANSI Modbus Point List Manual, create a CSV table with one row per point and columns for point name, function code, 4X address, register width in bits, data type, scale factor, offset, units, access, and IEC 61850 reference. Do not invent values. List fields that require confirmation from the manual."
7.3 Draft one read
"Generate beginner-friendly Python code for Windows that performs one Modbus TCP read of register 9000 (System Status Register 1) from a host at a specified IP address, using function code 03, and prints 
the MBAP header, the PDU, and the decoded 16-bit value. Keep the code in a single file and explain every step."
7.4 Validate with Wireshark
"I designed this Modbus TCP request and captured this traffic in Wireshark. Compare the designed bytes to the captured bytes field by field, identify any mismatch, and propose a corrected implementation. Keep the explanation suitable for a beginner."
7.5 Expand the backend
"Extend the read module so it can load a CSV point table, select any point, build the correct Modbus TCP request, parse the response, apply scaling, decode two-register 32-bit values with high-word-first ordering, and return both the raw and the engineering value. Keep the design modular and explain each function."
7.6 Implement a bounded write
"Using the REF620 ANSI 4X control-structure pattern, implement a Python function that issues an LED reset by writing the password registers, setting bit 0 of control register 1003, and committing through execute register 1000. Verify the confirmation register. Do not modify any circuit-breaker control bit."
7.7 Build the graphical interface
"Build a simple Windows graphical interface on top of the validated backend that lets the user choose a point, perform a single read or a periodic read, trigger the bounded LED reset or disturbance-recorder write, and inspect the request and response in both hexadecimal and decoded form. Do not add unrelated features."
7.8 Prompt refinement
State precisely what is wrong with the last output, for example wrong word order, missing MBAP transaction identifier, unhandled exception code, or incorrect scaling.
Restate the constraints: Windows host, Python, single file, function code 03 for reads, function code 06 or 16 for writes, and 
beginner-friendly
.
Ask the assistant to explain the problem before proposing new code.
Preserve known-correct sections of the code and revise only the failing part.
Use the ABB manuals and the Wireshark capture as final arbiters when the assistant makes assumptions.
